01 — WHAT ORION ACTUALLY IS

Orion is a persistent intelligence layer that sits between you and any AI model. It maintains memory, identity, and understanding across every conversation you have — regardless of which model, interface, or device you're using.

It is not an AI model. It does not generate text. It does not do inference. It is the memory and context system that makes any AI model aware of who you are, what you're working on, and how you prefer to interact.

What people think AI memory is: Model remembers things between sessions What actually happens (without Orion): Every session starts blank. Model knows nothing about you. Context files (CLAUDE.md) inject facts. That's briefing, not memory. What Orion does: Reads conversations from every source (Claude, Codex, Gemini, Telegram, phone) Extracts patterns, preferences, rules, project states Stores in its own memory (graph + knowledge index) Provides that context to any model that asks The brain persists. The model is interchangeable.
HONEST NOTE

The current "understanding" is mostly pattern matching and keyword extraction, not deep comprehension. The brain detects that you say "execute" a lot and tags you as "directive." It doesn't truly understand why you value execution over suggestion. Deeper understanding requires Orion's own trained model (in progress).

02 — DATA FLOW: WHAT HAPPENS WHEN YOU TALK

Every interaction with Orion follows this path, regardless of which interface you use:

YOU speak/type/call | v INTERFACE receives it (Codex terminal, Telegram bot, phone call, Claude Code, Gemini CLI) | v BRAIN processes it | +--> CLASSIFY: Is this a greeting, question, action, or command? | Uses keyword matching against known patterns | +--> RECALL: What does the brain already know about this topic? | Layer 1: Graph memory (microsecond tag lookup) | Layer 2: Knowledge index (BM25 text search) | +--> RESPOND: Generate response using available "fuel" (AI model) | Tries best available model in order of capability | +--> MEMORIZE: Should this interaction be remembered? | Classifies as ADD / UPDATE / DELETE / NOOP | Filters trivial messages ("hi", "thanks", "ok") | Checks for duplicates (similarity > 0.6 = skip) | Detects corrections ("actually", "no,", "wrong") | v STORED in brain's memory Graph memory: tagged, instant recall Knowledge index: BM25 searchable Conversation log: raw record for synthesis

Memory Classification (Mem0 Pattern)

Not everything gets stored. The brain classifies each interaction:

ADD

New information worth remembering. Stored in graph + index with extracted tags.

UPDATE

Correction or clarification of existing memory. Finds the old entry, replaces it.

DELETE

User said "forget that" or "scratch that." Finds and removes the matching memory.

NOOP

Not worth remembering. Trivial greetings, too short, or duplicate of existing memory.

WHAT THIS ACTUALLY DOES

Classification uses keyword matching, not semantic understanding. It looks for phrases like "actually", "never mind", "forget that" in the first 30 characters. Duplicate detection uses BM25 text similarity with a 0.6 threshold. This works for obvious cases but misses subtle corrections or context-dependent meaning.

03 — MEMORY SYSTEM

Two layers, both file-based, both running on the local machine:

GRAPH MEMORY (~/.orion/brain/graph_memory.json) Structure: { nodes: [ { id, content, type, confidence, tags, timestamp } ] } Lookup: Tag-based. Query "FORGE" → finds all nodes tagged "forge" Speed: Microseconds (it's a JSON file loaded in RAM) Size: Currently 30+ nodes Think of it as: A tagged notebook. Fast, structured, small. KNOWLEDGE INDEX (~/.orion/brain/knowledge_index.json) Structure: BM25 inverted index over compiled knowledge articles Lookup: Text search scored by term frequency / inverse document frequency Speed: Milliseconds Size: Grows as conversations are compiled into articles Think of it as: A search engine over everything the brain has learned. NOT a vector database. No embeddings. No semantic similarity. Qdrant (vector DB) exists on COMMAND but the portable brain uses file-based BM25 for zero-dependency operation.

How Recall Works

brain.remember("what is FORGE") | +--> Graph: scan all nodes, score by tag overlap with query tokens | Match: "FORGE travels with James" (tagged: forge, mobile, laptop) | Match: "RTX 4070 8GB VRAM" (tagged: forge, gpu, nvidia) | +--> Index: BM25 search for "what is FORGE" | Score documents by term frequency, return top 5 | +--> Combine results into <memory-context> block [graph] FORGE travels with James ... [index:0.84] FORGE is the primary workstation ...
LIMITATION

Graph recall is tag-based, not semantic. If you stored something about "my laptop" but search for "FORGE", it won't match unless "forge" is in the tags. BM25 is keyword-based — it matches words, not meaning. Searching "my computer" won't find documents about "FORGE" unless both words appear in the text.

04 — SYNTHESIS ENGINE

The synthesis engine reads conversations from all sources and produces structured understanding. This is what runs during each heartbeat cycle.

SOURCE READERS — Pull from native storage of each tool Claude Code: ~/.claude/projects/*/memory/*.md Codex: ~/.codex/sessions/ (JSON transcripts) Gemini: ~/.gemini/history/ (JSON sessions) Ollama: ~/.orion/brain/conversations/ (logged by brain) Letta: ~/.letta/agents/*/memory/ (MemFS files) Orion: ~/.orion/brain/conversations/ (all interfaces) All readers return the same format: { role: "user"|"assistant", text: "...", source: "codex", timestamp: ... } Currently reading: ~212 messages across all sources

What the Synthesis Engine Produces

User Model

What it tracks: Communication style (concise? directive?), frustrations (what triggers negative responses), expertise areas (frequent technical terms), values (what you praise), learned rules ("never X", "always Y"), correction patterns.

How it works: Counts word frequencies, detects directive language patterns, extracts capitalized terms as expertise, finds "never/always/don't" phrases as rules. Deduplicates by text similarity.

Stored at: ~/.orion/brain/user_model.json

Project State

What it tracks: Active projects (detected from capitalized words mentioned 2+ times), blocked items (negative signals near project names), decisions, next actions, cross-tool topic overlap.

How it works: Regex for ALL-CAPS words, quoted phrases. Checks surrounding context for "working on", "stuck", "decided" signals. Finds topics discussed in multiple tools.

Stored at: ~/.orion/brain/project_state.json

Self-Instructions

What it is: A markdown file the brain writes about itself. Contains the patterns it's discovered, rules it's extracted, and how it should weight different types of information.

How it evolves: Each synthesis cycle, new patterns are appended. Old sections get replaced. The brain literally rewrites how it processes data.

Stored at: ~/.orion/brain/self_instructions.md

Context Document

What it is: A 6-section briefing document that any AI model can load to instantly have full awareness of who you are and what's happening. Replaces static files like AGENTS.md.

Sections: Identity, User Understanding, Current State, Behavioral Rules, Cross-Tool Awareness, Recent Activity.

Generated on demand or cached for 5 minutes

THE GAP

Everything above is local analysis — pattern matching, keyword extraction, frequency counting. No neural network, no LLM, no deep understanding. The code has hooks for LLM enrichment (using a fuel model to produce richer analysis), but these require an AI model to run. The goal is for that model to be Orion's own fine-tuned model, not Claude or GPT. The fine-tuning pipeline is built (1,207 training pairs ready). The model hasn't been trained yet.

05 — HEARTBEAT

The heartbeat is a background thread that makes the brain think on its own schedule. Without it, the brain only processes data when someone explicitly calls it. With it, the brain is alive.

HEARTBEAT CYCLE (every 30 minutes) 1. READ Pull conversations from ALL sources Currently: Claude, Codex, Gemini, Ollama, Letta, Orion conversations ~212 messages across all sources 2. DETECT CHANGE Hash the current message set Compare to last cycle's hash If nothing changed → skip (don't waste processing) 3. ANALYZE Run user model builder (pattern matching on all messages) Run project state analyzer (detect active/blocked projects) Extract behavioral rules from corrections 4. EVOLVE Rewrite self_instructions.md based on new patterns The brain changes HOW it thinks, not just WHAT it knows 5. RECORD Log what changed to heartbeat.log Save cycle stats to heartbeat_state.json Track: changes detected, facts learned, duration REFLEX (immediate, not waiting for next cycle) Triggered when urgent input arrives: - Corrections ("never do X", "wrong", "stop") - Critical facts ("important", "emergency") - Strong signals that shouldn't wait 30 minutes Extracts rules immediately, saves to user model, then signals the heartbeat to run an early cycle.

Heartbeat State

Example heartbeat_state.json: { "cycle_count": 1, "total_facts_learned": 1, "last_source_hash": "fab8f5d0515fbf6a", "last_cycle_time": 1776240965, "cycle_history": [ { "cycle": 1, "changes": 5, "facts": 1, "duration_ms": 349 } ] } Example heartbeat.log: [2026-04-15 04:16:05] Cycle 1: new conversation data detected; user model confidence: 35% -> 50%; learned 1 new behavioral rule(s); new project(s) detected: ORION; self-instructions evolved
WHAT "ALIVE" ACTUALLY MEANS

The heartbeat is a timer running synthesis on a schedule. It's not consciousness. It's not thinking in the human sense. It reads data, runs pattern matching, updates stored models. But it does this autonomously — nobody has to trigger it. And it processes data from ALL interfaces simultaneously, which no single AI model can do. That cross-source awareness is the real value.

06 — MCP SERVER: HOW AI MODELS CONNECT

09 — WHAT NO OTHER SYSTEM DOES

This section documents what Orion does that no existing AI memory system, agent framework, or personal AI product currently achieves. Verified through comprehensive research across Mem0, Letta, Zep, Khoj, Hindsight, Second Me, LangMem, and 20+ other systems (April 2026).

1. UNIFIED CROSS-TOOL AWARENESS

Every AI tool — Claude Code, Codex, Gemini, Ollama, Letta — stores its own conversation history in its own format in its own location. None of them can see what the others are doing. Claude doesn't know what you told Codex. Gemini doesn't know what Ollama said.

Orion reads from ALL of them simultaneously. It has 9 source readers that parse 6 different formats (Claude JSONL, Codex JSONL, Gemini JSON, Letta MemFS markdown, Ollama conversation logs, and its own conversation records). Every heartbeat cycle, it pulls from all sources and builds understanding across the full picture.

No other system does this. Mem0 stores memory but only from one application at a time. Letta manages its own memory but can't read Claude's sessions. Zep tracks conversation history within its own framework. None of them reach into other tools' storage and unify it.

2. CROSS-MODEL MEMORY

Store a fact through Codex. Recall it through Gemini. Both models call the same brain through MCP tools. The memory lives in Orion, not in any model. Verified working — Codex stored "James likes key lime pie" via orion_memorize, Gemini can recall it via orion_remember.

This is different from each model having its own memory. This is one memory that multiple models share. Switch models mid-thought and your context follows.

3. EVERY COMMUNICATION POINT → ONE BRAIN

AI models connect via MCP. Phone calls connect via Telnyx + HTTP. Telegram connects via bot API + HTTP. iMessage connects via AppleScript bridge. Terminal conversations connect directly. Web dashboard connects via webhooks. All feed the same brain. All draw from the same memory.

No other personal AI system unifies AI model conversations, phone calls, messaging apps, and terminal sessions into one persistent brain. These are typically separate systems with separate memories.

4. PHYSICAL PORTABILITY

The brain is JSON files on disk. The code is pure Python with zero dependencies. Copy the ~/.orion/ folder to a USB drive, plug it into any computer with Python installed, and the brain works. No database server. No cloud account. No API keys required for the brain itself.

No other system achieves this. Mem0 requires a vector database. Letta requires its server runtime. Zep requires Neo4j. Orion's brain is files you can carry in your pocket.

5. SELF-EVOLVING INSTRUCTIONS

The brain doesn't just store data — it rewrites its own operating rules. The self_instructions.md file is maintained by the synthesis engine. Each cycle, it analyzes what patterns it found, what rules the user expressed, what tools are used most, and rewrites the instructions that control how future synthesis runs. The brain changes how it thinks, not just what it knows.

LangMem has "procedural memory" (agents updating their own prompts) but it's manual. Orion's self-evolution is automatic and runs every heartbeat cycle.

6. HYPER INGESTION

Run python orion_brain_portable.py --hyper and the brain reads every conversation you've ever had across all AI tools, memory files, knowledge documents, and context files. In testing: 1,070 messages from 9 sources, analyzed in 741 milliseconds. Confidence goes from 0% to 100% in one shot.

This is how a new Orion installation bootstraps itself. Install Orion, point it at your data, run hyper, and the brain immediately knows who you are from your existing conversation history. No manual setup. No telling it facts one at a time.

10 — HOW THIS WAS BUILT: AI BUILDING AI

Orion was built using AI tools. This is not hidden, not downplayed, and not something we're ashamed of. It's part of the story, and understanding it matters.

What AI Did

DEVELOPMENT TOOLS

Claude Code (Anthropic) was the primary development tool. It wrote Python code, debugged protocol issues, conducted research, designed architecture, and documented the system. It is a $200/month AI coding assistant used the same way an engineer uses an IDE.

Codex (OpenAI) was used for testing cross-model memory and validating the MCP server. It was the first non-Claude model to successfully connect to the Orion brain.

Research agents — 5 parallel AI agents conducted deep research across Ollama internals, universal AI interfaces, persistent identity systems, learning architectures, and competing products. Their findings informed the architecture but are not part of the running product.

OPEN-SOURCE BASE MODELS

The fine-tuning pipeline starts with open-source models like Qwen3 or Mistral — freely available, downloadable, modifiable. These are raw materials, not someone else's product. Orion trains a personal overlay (LoRA adapter) on top of these base models using the user's own conversation data on the user's own hardware.

This is standard practice across the entire AI industry. Meta released Llama for anyone to use. Alibaba released Qwen. Mistral AI released Mistral. These are building blocks, like open-source libraries in software development.

What AI Did NOT Do

THE HUMAN CONTRIBUTION

The vision — "The memory IS the intelligence. The model is fuel." This philosophy that a brain's identity lives in its memory, not in any particular model, is the foundational insight behind everything Orion does. No AI model produced this idea.

The architecture decisions — Every major design choice was human-directed. One brain serving all interfaces. Physical portability on a USB drive. Independence from any single AI provider. The heartbeat concept. Cross-model memory. The refusal to just wrap someone else's product and call it new. These decisions shaped the entire system.

The product direction — What to build, why it matters, who it's for, what's worth doing and what's not. AI tools suggest options. A human chose which options to pursue and which to reject. When AI suggested "there are only two ways to do this," the response was "there are infinite combinations" — and that philosophy drove the research and architecture in directions no AI would have taken on its own.

The quality standard — "Never claim something works without the user testing it." "No demo mode — everything must be functional." "No marketing fluff." These standards forced honest documentation, real testing, and actual functionality over impressive-sounding demos that don't work.

The persistence — Months of building. Sessions that went late into the night. Debugging protocol mismatches, fixing encoding crashes, rewriting architectures when they didn't work. Starting over when a direction was wrong. Continuing when nothing worked the first time. AI doesn't persist through frustration. A person does.

12 — OWN INFERENCE ENGINE: NO DEPENDENCIES

A critical product requirement: Orion must not depend on Ollama, vLLM, or any third-party runtime to function. The inference engine — the thing that runs the trained model — must be part of Orion itself.

What We Discovered

THE PROBLEM Orion trains its own model (LoRA adapter on open-source base) But running that model requires an inference engine Ollama is one engine — but it's a separate install, a dependency Users shouldn't need to know what Ollama is THE SOLUTION llama.cpp is open source (MIT license) llama-cpp-python embeds the engine directly in Python One process = brain + model + memory + heartbeat No separate service. No HTTP overhead. No external dependency. pip install llama-cpp-python from llama_cpp import Llama llm = Llama(model_path="orion.gguf", n_gpu_layers=35) output = llm("Hello sir", max_tokens=128) That's the entire inference engine embedded in Orion's process.

How Inference Actually Works (The Real Mechanics)

STEP BY STEP — what happens when you talk to the trained model: 1. Your text arrives: "What's running on COMMAND?" 2. TOKENIZE: Convert text to numbers "What's" → 1724 "running" → 2846 "on" → 373 "COMMAND" → 24118 "?" → 30 Now it's [1724, 2846, 373, 24118, 30] 3. LOAD CONTEXT: Feed tokens through the model Each token passes through 32 transformer layers Each layer: multiply by weight matrices, compute attention, multiply again 7 billion parameters = 7 billion multiplication operations per layer This is where the GPU earns its keep 4. ATTENTION: "Which previous tokens matter for the next word?" The model computes relationships between every token pair "COMMAND" gets high attention from "running" and "What's" This is how context works — the model knows the question is about COMMAND 5. PREDICT: Final layer outputs probability for every possible next word "Docker" = 12% "The" = 8% "Currently" = 7% "Sir" = 15% ... Pick one (sampling with temperature) 6. REPEAT: Add the chosen token, go back to step 3 Until it generates a stop token or hits max length THE LORA DIFFERENCE The base model (Mistral 7B) knows general language The LoRA adapter (42M params trained on YOUR conversations) adjusts the weights At step 5, "sir" gets a higher probability because the adapter learned that pattern The model doesn't READ a rule saying "say sir" — it's in the weights

Persistent KV Cache — The 136x Speedup

WITHOUT KV CACHE (current behavior) Every conversation starts cold Model processes the 2000-token system prompt from scratch Takes 15+ seconds before it can respond to your actual question WITH PERSISTENT KV CACHE First time: process system prompt, save the computed key/value tensors to disk (~50MB file) Every time after: load from disk in <1 second Model instantly has full context without recomputing Benchmark: 172s cold → 1.3s hot = 136x faster startup For Orion: the identity, rules, user model, and project state are pre-computed once and loaded from cache every conversation. The brain feels instant.

13 — THREE-TIER ARCHITECTURE

Not every query needs a 7-billion-parameter model. Most daily interactions are simple — "check status", "what's on my schedule", "remember this". The architecture uses the right amount of compute for each task.

TIER 1 — REFLEX BRAIN (always on) Hardware: Raspberry Pi 5 (8GB RAM, ~5W power) Model: BitNet 2B or Gemma 1B (1.58-bit or Q4) Speed: 8-15 tokens/second Memory: <2.5GB RAM Role: Dispatch commands, status checks, quick chat Handles: ~60% of daily queries Startup: <30 seconds from boot Always on. Always listening. Costs pennies in electricity. TIER 2 — LOCAL INTELLIGENCE (on demand) Hardware: FORGE RTX 4070 (8GB VRAM) Model: Mistral 7B + Orion LoRA (Q4_K_M, ~5GB) Speed: 40-60 tokens/second (with speculative decoding) Role: Reasoning, planning, code, complex tasks Handles: ~35% of queries (escalated from Tier 1) Boost: Speculative decoding with 1B draft model = 1.5-2x speedup Real thinking. No API calls. Your hardware, your model. TIER 3 — HEAVY COMPUTE (rare) Hardware: Cloud (Claude API) Model: Claude Opus or equivalent Speed: Cloud speeds Role: Complex multi-step analysis, large context reasoning Handles: ~5% of queries (things Tier 2 can't solve) The only tier that uses external AI. Optional. Not required.
HOW ROUTING WORKS Query arrives (any interface: Telegram, phone, terminal, MCP) | v TIER 1 (Pi): Can I handle this? Simple dispatch, status check, greeting → respond directly Complex reasoning needed → escalate | v TIER 2 (FORGE): Can I handle this? Reasoning, code, planning → respond Exceeds local model capability → escalate | v TIER 3 (Cloud): Handle the extraordinary Same brain at every tier. Same memory. Same identity.
